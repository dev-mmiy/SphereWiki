# SphereWiki — MVP & Roadmap

> Companion to [`PRODUCT.md`](PRODUCT.md) (requirements + decisions AD-1…AD-8) and [`../CLAUDE.md`](../CLAUDE.md) (loop & architecture). This file defines the MVP scope and the milestone path to it.

## MVP definition

**Goal:** internal / small-team **dogfooding** — validate the core value through daily real use, with a deliberately lower external-polish bar.

**Central hypothesis:** *An AI that automatically grows a shared team knowledge base — auto-linking, auto-tagging, and lightly reorganizing notes on save, with every change versioned and revertible — is valuable enough that a small team keeps using it.*

**Success criteria** (exit the MVP phase when):
- A small team uses SphereWiki as its real knowledge base for ≥ 3–4 weeks.
- On-save AI suggestions (links/tags) are kept far more often than reverted (target ≥ ~70% kept).
- The note graph measurably grows (links/notes per week), with AI contribution visible.
- Users trust AI edits *because* every change is attributed and one-click revertible (low "AI broke my note" incidents).
- Async multi-user sharing works with no lost edits (CRDT convergence holds in practice).

**Quality bar:** *data* reliability (no lost edits, isolation holds, revert always works) is non-negotiable; UI polish, feature breadth, and scale are not.

## In scope (MVP)

| Area | MVP includes |
|---|---|
| Platform | Desktop only (Tauri 2; build all-OS, test on dev OS) |
| Notes | Markdown + frontmatter, `[[wikilinks]]`, backlinks, tags, basic graph view |
| Editor | CodeMirror 6 source editor bound to a per-note Yjs `Y.Text` |
| Collaboration | **Async multi-user** — several people share a workspace; edits merge (Yjs). No real-time cursors/presence yet |
| Sync | Hocuspocus super-peer (server-readable); Cloud SQL + GCS persistence |
| Accounts | WorkOS: accounts (required), organization, workspaces, roles; isolation enforced (RLS + scope) |
| Versioning | Engine-agnostic history layer: timeline, diff, revert, human/AI attribution |
| **AI (core)** | On-save agent (server-side CRDT peer, BYO Claude key): **auto-link + auto-tag**; AI edits versioned, attributed, revertible |
| RAG | Embeddings (e5-small ONNX, desktop + server), pgvector/DuckDB index, basic Q&A over a workspace |

## Out of scope (deferred — see Post-MVP)

Real-time co-editing (cursors/presence) · Web client · Content ingestion (PDF/Office/image OCR) · MCP server · Shared-workspace RAG-inclusion governance UI (MVP starts with project workspaces + at most one shared) · Billing/Stripe · E2E + per-org trusted worker · Mobile · SSO/SCIM · advanced reorganization.

> Architectural invariants still hold for deferred items (isolation, engine-agnostic versioning, control/data-plane split), so deferring is additive — not a future rewrite.

## Milestones to MVP

Each milestone ends green on all Loop gates (`pnpm verify`) and meets its acceptance criteria.

**M0 — Foundations & the Loop.** pnpm monorepo; packages `shared` / `desktop` / `server` / `ai` as stubs; wire the stable command aliases (`pnpm verify` / `typecheck` / `lint` / `test` / `build`, `pnpm reindex`); first test green.
*Done when:* `pnpm verify` passes on the stub repo and the loop is runnable end-to-end.

**M1 — `shared` core (offline, pure).** Markdown/frontmatter parsing; wikilink/backlink/graph; the **CRDT adapter (Yjs)** + **engine-agnostic version layer** (snapshot / restore / diff / revert / attribution contract); shared types.
*Done when:* unit-tested, platform-free, with strong coverage on link integrity and version revert.

**M2a — Desktop editor, editor-first (done).** Vite + React; CodeMirror 6 ↔ per-note Yjs; multi-note in-memory `Vault`; wikilink/backlink/graph navigation; **history/diff/revert UI**. Runtime-verified via jsdom render + hook tests; runs today as a web build (`pnpm dev`).
*Done:* ✅ a single user edits, navigates links, and uses history/revert; `pnpm verify` green.

**M2b — Native shell & persistence (deferred — needs Rust/Tauri).** Tauri 2 wrap; **on-disk Markdown vault** (file-backed `Vault` behind the existing seam); **DuckDB** local index (FTS/graph/vectors) + idempotent `reindex`. Split out of the editor-first slice because it needs the Rust toolchain (heavy on a slow link) — best done in a dedicated session.
*Done when:* the desktop app reads/writes a real `.md` vault offline and search works.

**M3a — Sync, persistence & auth foundations (done, local-testable).** The super-peer **sync seam** (in-memory hub: two-client convergence, server-readable authoritative replica, no echo); the **persistence seam** (durable per-room replica, restored across restart, per-room isolation); the **auth seam** (sessions/roles + `can`/`roleFor`, non-member = no access). All pure, swappable, unit-tested — no external services.
*Done:* ✅ convergence, durability, and permission checks proven in tests; `pnpm verify` green.

**M3b — Real WebSocket transport + durable storage + app sync (done, local) & cloud control plane (deferred — credentials).** ✅ **Hocuspocus WebSocket** super-peer (`apps/server`) — two real clients converge over actual sockets (integration-tested). ✅ **Durable file persistence** (`createFilePersistence`) — one hashed, fixed-length file per room under `SPHEREWIKI_DATA_DIR`, written atomically (temp + rename); room state survives a server restart; the server is runnable via `pnpm dev:server`. ✅ **Desktop wired to the super-peer** — set `VITE_SYNC_URL` and the active note syncs live per room (`workspaceId/noteId`). When syncing the server is authoritative for the room and the client hydrates from it; the writeback is **hydration-guarded** so an un-hydrated (offline / fast-switch) doc can never overwrite the Markdown vault with empty content. Roundtrip integration test proves convergence (incl. concurrent edits) + restore-after-restart over real sockets; the desktop seed-vs-sync + no-clobber logic is unit-tested via an injected transport. **Deferred (need credentials):** **WorkOS** AuthKit (auth seam); **Cloud SQL + GCS** (persistence seam) + Postgres RLS for DB-level isolation.
*Remaining done-when:* desktop/web sync against the *deployed* server with real accounts, and cross-workspace access is provably blocked at the DB.

> **Storage & sync follow-ups (local, no credentials), surfaced by the S1/S2 review:**
> - ✅ **Durable local vault (S3a, done):** the desktop vault persists to `localStorage` (`createLocalStorageVault`, behind the `Vault` seam), so the note list + Markdown survive a reload offline; it degrades to in-memory if storage is unavailable. Note ids stay stable across reloads so sync room names don't drift.
> - ✅ **Offline-first for *synced* rooms (S3b, done):** a synced room now caches its CRDT state locally via the `ConnectLocalPersistence` seam (real impl `connectLocalPersistence`, backed by **y-indexeddb**). On open the last-synced state loads from IndexedDB so the editor is **readable offline with no server**; the super-peer then merges live edits on top of the same Yjs doc (both are CRDT updates, so they converge). The vault is **never clobbered**: in sync mode an empty doc is never written back over real Markdown — a single empty-doc write guard covers the local-cache path, the server `onHydrated` path (the real super-peer fires `synced` even for an empty room), and the cleanup write-back. Once the authoritative state has arrived the room is hydrated, so the first real edit to an empty synced room persists. Seeding stays **server-authoritative** for synced rooms (the client never seeds a synced room from the local vault), so two clients can't double-seed into garbled text. Verified with injected fakes (hook logic, incl. a Red→Green regression test for the empty-room clobber) and a real `fake-indexeddb` round-trip (content survives reopen). *Remaining:* an **uncached** empty synced room shows an empty editor until the server (or a prior local cache) provides content — the vault's seed Markdown is preserved but not displayed, because synced docs aren't seeded client-side; showing/seeding a brand-new room offline needs server-arbitrated seed-once (M3b).
> - **Whole-vault sync (S4):** ✅ *S4a (done)* — an engine-agnostic **note-registry CRDT** (`CrdtRegistry` / `openYjsRegistry`, a per-workspace map of note id → `{title}`) so creating / renaming / removing a note can propagate to peers, distinct from each note's body `CrdtNote`. Convergent + idempotent + remote-flagged, with a read-boundary guard that drops malformed remote entries, all unit-tested.
>   - ✅ *S4b-1 (done)* — **collision-resistant note ids:** new notes get `crypto.randomUUID` ids (was per-client `n1,n2,…`), seeds included; legacy `n*` ids migrate to UUIDs on load before any room is joined. So two peers can never mint the same id → the registry key and the body room `${ws}/${id}` can't collide. Plus `Vault.ensure(id,title,body?)` (insert-if-absent, never overwrites a body).
>   - ✅ *S4b-2 (done)* — **the note list converges across peers:** the registry syncs over a workspace-level room `${ws}/__registry__` via dedicated `ConnectRegistry` / `ConnectRegistryPersistence` seams (mirroring the body seams; IndexedDB cache + offline), in a hook effect keyed on the workspace (not the active note). A note created on one peer appears on the others; reconcile is **registry→vault add-only** (never removes a local note for being absent from a remote/empty registry), the list is union-additive (registry title wins for display), and seeding stays **server-authoritative** (clients never bulk-publish their local seeds → no double-seed). A freshly created note's body is seeded into its own synced doc (unique id, single author — safe), so the creator sees it and peers receive it. Adversarial-reviewed (data-loss/double-seed/collision/lifecycle); 0 real findings.
>   - ✅ *S4c (done)* — **revertible synced delete:** a note delete is a **registry soft-delete tombstone** (`RegistryEntry.deleted`), an additive/convergent registry edit that hides the note from the list across peers but **never erases its Markdown body** — so it is restorable (a "Trash" with Restore in the UI) and a peer delete can't silently destroy local work. Reconcile stays add-only (a tombstone for a note this replica never had isn't resurrected into its trash); deleting the active note (here or remotely) moves to a visible note, and deleting the *last* visible note hides the editor rather than leaving it editable on a trashed note. Malformed remote tombstones are coerced (only strict `true` tombstones). Adversarial-reviewed; the one real finding (editable editor on the last-deleted note) was fixed + regression-tested. *Deferred to M3b/S4d:* promoting local-only/seed notes into a shared registry (server-arbitrated seed-once), true single-socket multiplexing, and duplicate-`Home` dedup.
>   - ✅ *S4d (done)* — **link-integrity-preserving rename:** renaming a note is the first feature to exercise the **link-integrity invariant**. Because wikilinks resolve by *title*, rename updates the title **and atomically repoints every `[[old title]]` backlink** (preserving `|alias` / `#anchor`) so no dangling link is left behind. The shared primitive `renameWikiLinkTargets` is pure and literal-matched (regex-special titles like `C++` are safe). The title change goes to the **registry** (the title authority; converges to peers via LWW), the **open** note's body rewrite rides its CRDT doc (synced, attributed, revertible), other notes are rewritten in the vault, and the vault's own title metadata is kept in step (new `Vault.rename` seam — a future file-backed vault renames the `.md`). Renaming to a title already used by another visible note is refused (duplicate titles would make `[[title]]` ambiguous). A per-note ✎ control (role-gated) drives it via prompt. Adversarial-reviewed (4 dimensions × verify); confirmed fixes landed: the active-id is now skipped *unconditionally* in the vault loop so a mid-hydration doc can't clobber the rewrite, the duplicate-title guard, and vault-title consistency; tests strengthened (all-backlinkers, peer title convergence, two-rename no-drift, functional viewer-block). *Documented bounds (non-destructive, fully correct local-first/single-writer):* the title is a registry-level edit reversed by renaming back (not in per-note body history, like delete/restore) — so a body revert across a rename can surface stale link text; and in sync mode a **non-open** note's body rewrite (and a **concurrent same-note** rename) stays local until reopened, pending the deferred multi-doc sync.
> - ✅ **Persistence compaction (done):** the version layer kept a **full self-contained restore point per commit** (`snapshot()` = `Y.encodeStateAsUpdate(liveDoc)`). Because the live editing doc runs **`gc:false`** (AD-4 history substrate — tombstones retained), that raw encoding carries *every deleted character*, so a churn-heavy note's N versions stored N tombstone-bloated copies — unbounded growth. `snapshot()` now round-trips the current state through a **`gc:true`** doc before encoding, so each stored restore point carries only the version's **visible content** (tombstones collapse to tiny GC structs): a 5 000-char insert-then-delete snapshot drops from ~5 KB to ~40 B (~250×), bounded by visible content not edit churn. The snapshot stays **self-contained** (applying it to a fresh doc reconstructs the exact text — verified across a churn-heavy history), so revert/diff/open are unchanged. Scope is deliberately tight: only the version-layer restore points are compacted — the live doc's `gc:false` (AD-4, "tombstone memory mitigated by per-note granularity") and the sync `encodeState()` path are **untouched**, so CRDT convergence and the future `Y.snapshot`/`PermanentUserData` history path are preserved. *Deferred:* the live-doc / IndexedDB / super-peer state still grows with churn (accepted per AD-4); true delta-encoded history (`Y.snapshot` + `createDocFromSnapshot`) is a larger AD-4-aligned step — delta-chaining the current store is unsafe because a note's doc is re-seeded (fresh client id) on every note-switch, so committed states don't share CRDT lineage.
> - ✅ **Basic graph view (done):** the MVP "basic graph view" (Notes scope) now ships. A pure shared primitive `buildGraphModel(notes, linkGraph)` turns the workspace's notes + link graph into a renderable node/edge model: one node per visible note (isolated notes included), and an edge for every `[[wikilink]]` whose target **title** resolves to a real note's **id** — **dangling** links (no matching note) and **self-links** are dropped and duplicate edges collapse to one, so the graph shows only real connections. It's derived from Markdown (rebuildable, idempotent) and **workspace-scoped by construction** — it only ever contains the notes handed in, so it can't reach across workspaces. The desktop `GraphView` renders it as SVG with a **deterministic circular layout** (no physics engine — cheap, stable, unit-testable): each node is an accessible button (`role="button"` + `aria-label`, keyboard-activatable) that navigates on click; the active note is highlighted; presentation uses SVG attributes so it renders without a stylesheet. The hook exposes the model as `graph` (a `useMemo` over the same note list + link graph, so it tracks every body/title change), and it's wired into the workspace below the Tags panel. Unit-tested (model: resolution/dangling/self-link/dedup; view: render/click-navigate/Enter/active-marking/edge-count) + an integration test (click a graph node → the active note moves).
> - ◐ **Room-level authorization — server enforcement seam (done); token issuance deferred.** The super-peer now has an injectable `RoomAuthorizer` (`createSyncServer({ authorize })`): when set, it wires Hocuspocus `onAuthenticate` so every join must present a token the authorizer accepts **for the exact room** it is joining, or the connection is rejected (the client never syncs). This is the transport-level enforcement point that hardens **project isolation** / **permissions** beneath room naming (control plane stays server-authoritative — a peer can't reach a workspace's room by guessing its name). It defaults **open** when no authorizer is configured, so local/dev sync stays zero-config and the existing behaviour/tests are unchanged. Verified over real sockets: an accepted room-scoped token joins and syncs; a denied or empty token is rejected; and a token granted for one room can't join another. *Deferred to M3b (credentials):* the **WorkOS-issued, workspace-scoped token** the real authorizer verifies, and threading that token from the desktop through the connect seams (`ServerSyncOptions.token` → provider) — the production super-peer leaves `authorize` unset until then.

**Known M4a limitations (folded into M4b)** — surfaced by the M4a adversarial review; each is bounded and additive, behind the existing seams:
- *Runtime read-only shared indexes:* `ReadonlyVectorIndex` is read-only at the type level only; the real backends (and a frozen projection) make it enforced at runtime.
- *Shared-bridge type enforcement:* the RAG retriever trusts the caller that a passed `shared` index is a `shared`-type workspace; M4b carries workspace `type` and adds Postgres RLS so "never another project" is checked, not conventional.
- *Citation id namespacing:* `readBody` resolves by bare `NoteId`; M4b namespaces ids per workspace so cross-workspace id collision can't source citation text out of scope.
- *Title-metadata staleness:* the embedding/staleness key is body-only (deliberately, so tag edits don't churn the vector); a note rename leaves a stale stored title until re-embed. M4b refreshes title metadata independently.
- *CJK / space-less auto-link:* the heuristic's letter-class word boundaries rarely fire for scripts without inter-word spaces; the M4b Claude suggester (or a segmenter) handles these.
- *Frontmatter canonicalization:* an AI tag edit re-serializes frontmatter to canonical YAML style; M4b can splice the tags region to preserve human formatting if needed.

**M4a — On-save AI agent + RAG, local-testable (done).** The `@spherewiki/ai` package: a content-addressed **embedding seam** + deterministic in-memory embedder; a **per-workspace vector index** sealed to one workspace by construction (cross-workspace access is unrepresentable) with a read-only view as the only shared-workspace bridge; a no-LLM **heuristic suggester** (auto-link + auto-tag) with pure, idempotent application; a **scoped RAG retriever** (project + opt-in read-only shared) and a deterministic extractive answerer; the **on-save agent** that applies AI edits as a CRDT peer via `setText({kind:'ai'})` + a committed Version — attributed, revertible, merge-safe, permission- and autonomy-gated; and an idempotent **reindex** that rebuilds the index from Markdown alone. All seven domain invariants are exercised in tests (incl. type-level isolation guards). No credentials/downloads.
*Done:* ✅ on save, AI links/tags appear as attributed, revertible edits; retrieval is workspace-scoped and cited; `pnpm verify` green. The agent is **wired into the desktop UI** — an "Organize with AI" action runs it on the active note (gated by role, in-flight-guarded, edits land in the history panel as revertible AI versions) using the local heuristic suggester + embedder, and a RAG **"Ask the workspace"** panel returns cited answers (idempotent reindex → scoped retrieval → extractive answer). Split out of M4 because the real backends need a model download + a Claude key.

**M4b — Real AI backends + wiring (deferred — credentials/download).** Real **multilingual-e5-small (ONNX)** embedder (model download + forced reindex), **pgvector** (server) / **DuckDB** (desktop) vector backends with Postgres RLS beneath the by-construction isolation, the **Claude-backed** suggester + answerer (BYO key via Secret Manager), and the **server-side agent as a Hocuspocus CRDT peer** — all behind the M4a seams, unchanged (the desktop "Organize with AI" action is already wired and will simply pick up the real providers; the agent already guards against a mid-run note switch so the real async backends are safe).
*Done when:* saving a note on the deployed stack triggers AI links/tags as attributed, revertible edits, and Q&A returns cited answers scoped to the workspace.

**M5 — Dogfood hardening.** Reliability (no lost edits, revert always works), onboarding, and **success-criteria instrumentation** (kept-vs-reverted, graph growth).
*Done when:* the team runs it daily and the success criteria above are measurable. **← MVP complete.**

## Post-MVP (indicative order)

1. **Real-time co-editing** — presence/cursors on the existing Yjs base.
2. **Web client** — via the super-peer gateway.
3. **Content ingestion** — PDF/Office, then image OCR/captioning.
4. **Multi-project depth** — shared-workspace governance UI, more workspace attributes.
5. **MCP server** — expose the vault to external agents.
6. **Billing** — Stripe subscription (customer = org).
7. **Privacy endgame** — E2E + per-org trusted worker (AD-1).
8. **Later** — mobile, SSO/SCIM, AlloyDB if scale demands.
